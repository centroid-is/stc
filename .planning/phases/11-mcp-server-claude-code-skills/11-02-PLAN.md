  <files>.claude/skills/st-generate.md, .claude/skills/st-validate.md, .claude/skills/st-test.md, .claude/skills/st-emit.md, .claude/skills/st-review.md</files>
  <action>
    Create each skill markdown file with the following structure. Each skill must include: purpose, when to use, step-by-step workflow with exact CLI commands, expected output interpretation, and error handling.

    **st-generate.md** (SKIL-01):
    Purpose: Generate IEC 61131-3 Structured Text code from natural language descriptions.
    Workflow:
    1. Understand the user's intent (PLC program, function block, function, etc.)
    2. Generate ST code following IEC 61131-3 Ed.3 conventions:
       - Use UPPERCASE keywords (PROGRAM, FUNCTION_BLOCK, VAR, END_VAR, etc.)
       - Use 4-space indentation
       - Declare all variables in VAR blocks before use
       - Use meaningful PascalCase names for POUs, camelCase for variables
    3. Write generated code to a .st file
    4. Validate immediately by running: `stc parse <file> --format json`
    5. If parse errors, fix them and re-validate
    6. Run type checking: `stc check <file> --format json`
    7. If type errors, fix and re-check
    Key patterns: PROGRAM for main logic, FUNCTION_BLOCK for reusable stateful components, FUNCTION for pure computations. Standard FBs available: TON, TOF, TP, CTU, CTD, R_TRIG, F_TRIG, SR, RS. Standard functions: ABS, SQRT, MIN, MAX, LEN, CONCAT, LEFT, RIGHT, MID, FIND, INT_TO_REAL, REAL_TO_INT, etc.

    **st-validate.md** (SKIL-02):
    Purpose: Validate ST code through the full parse + check + lint pipeline.
    Workflow:
    1. Parse: `stc parse <file> --format json` - Check for syntax errors
    2. Check: `stc check <file> --format json` - Check for type errors, undeclared variables, unused variables
    3. Lint: `stc lint <file> --format json` - Check for coding standard violations
    4. Interpret results: errors (must fix), warnings (should fix), info (consider)
    5. For vendor-specific validation, add: `stc check <file> --vendor beckhoff --format json`
    Pipeline exit codes: 0 = clean, 1 = errors found. Lint warnings are exit 0.
    Always run all 3 stages even if earlier stages have warnings -- only stop on parse errors.

    **st-test.md** (SKIL-03):
    Purpose: Write and run ST unit tests with the stc test framework.
    Workflow for writing tests:
    1. Create a *_test.st file alongside the source file
    2. Use TEST_CASE 'name' / END_TEST_CASE blocks
    3. Declare test variables in VAR blocks inside TEST_CASE
    4. Use assertions: ASSERT_TRUE(cond), ASSERT_FALSE(cond), ASSERT_EQ(actual, expected), ASSERT_NEAR(actual, expected, tolerance)
    5. For timer/FB testing: declare FB instances in VAR, use ADVANCE_TIME(T#100ms) to simulate time passing
    6. For I/O mocking: set input variables directly before calling FB logic
    Workflow for running tests:
    1. Run: `stc test <dir> --format json` (or --format junit for CI)
    2. Interpret: passed/failed/error counts, assertion positions
    3. Fix failing tests based on assertion messages with file:line:col
    Test file naming: <source>_test.st (e.g., motor_control_test.st for motor_control.st)

    **st-emit.md** (SKIL-04):
    Purpose: Emit vendor-specific Structured Text from portable ST source.
    Workflow:
    1. Validate source first: `stc check <file> --format json`
    2. Emit for target vendor: `stc emit <file> --target <vendor> --format json`
       - beckhoff: Full CODESYS OOP, pointers, references, 64-bit types
       - schneider: No OOP, no pointers/references
       - portable: Safest cross-vendor subset (no OOP, no pointers, no 64-bit)
    3. Review emitted code for vendor-specific pragma/attribute differences
    4. For round-trip verification: parse emitted code back to confirm stability
    Common pattern: Write portable ST, emit for each target vendor separately.

    **st-review.md** (SKIL-05):
    Purpose: Review ST code against IEC 61131-3 best practices and PLCopen guidelines.
    Workflow:
    1. Run lint: `stc lint <file> --format json` for automated checks
    2. Review lint results covering:
       - Magic numbers (use named constants instead)
       - Nesting depth (max 3 levels recommended)
       - POU length (max 200 lines recommended)
       - Missing return types on functions
       - Naming conventions (PascalCase for POUs, configurable)
    3. Manual review checklist (beyond linter):
       - Are VAR_INPUT/VAR_OUTPUT clearly defined for FB interfaces?
       - Are timer presets using constants, not magic numbers?
       - Is error handling present for edge cases?
       - Are comments present for non-obvious logic?
       - Could any logic be extracted into reusable FBs?
    4. Run type check with vendor target if deploying: `stc check <file> --vendor <target> --format json`
    5. Summarize findings with severity and actionable fix suggestions

    All skills must:
    - Reference exact `stc` CLI commands with --format json
    - Include the expected JSON output structure interpretation
    - Handle error cases (file not found, parse errors, etc.)
    - Be self-contained (no cross-skill dependencies required)
